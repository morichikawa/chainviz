# chainviz 開発ガイド

Docker 上の Ethereum ノード群を Miro 風の無限キャンバスでリアルタイム可視化する
学習・開発支援アプリ。構想・機能・決定事項は `docs/CONCEPT.md` を正とする。
実装の正確な記述は `docs/ARCHITECTURE.md`（設計フェーズで作成）に置き、
コード変更時は sync-docs スキルで docs/ との齟齬を確認する。

作業の進行順序は `docs/PLAN.md` に従う（ステップ・サブ項目がチェックボックスで
管理されている。作業を完了したらその場でチェックを付け、未チェックの中で
最も上にある項目が次にやることになる）。

## 設計原則（コードを書くとき常に守る）

- **境界を崩さない**: 「ノード環境（被可視化対象）→ Collector → GUI フロント」の
  一方向の依存を守る。フロントは Docker やノードの API に直接触れない。
  観察も操作も必ず Collector を経由する
- **ChainAdapter 境界**: チェーン固有のロジック（RPC メソッド名、レスポンスの
  正規化）は Adapter 実装の中に閉じ込める。ワールドステートのスキーマや
  フロントのコードに `eth_getLogs` のようなチェーン固有の語彙を漏らさない
- **チェーンプロファイル単位で増やす**: 新チェーン対応は「ノード環境テンプレート・
  ChainAdapter・フロント表現セット」の3点セットを新規追加する作業。
  既存プロファイルのコードに手を入れて分岐を足す方向に行かない
- **データとコードの分離**: 用語集（`glossary/`）・チェーンプロファイル設定の
  ような更新頻度が高いコンテンツはデータファイル（YAML/JSON）に置き、
  コードはそれを読むだけにする

## フォルダ構成の方針

具体的な構成は `docs/ARCHITECTURE.md` で確定させる。従う原則:

- TypeScript で統一したモノレポ。少なくとも collector / frontend /
  共有スキーマ（ワールドステートの型定義）をパッケージとして分離し、
  型を共有する
- 技術レイヤー（controllers/, utils/ など）ではなく、ドメイン・機能単位で
  モジュールを切る（例: collector 内は adapters/, world-state/, docker/ など）
- 1 ファイル 1 責務。ファイルが肥大化してきたら分割を検討する
- 循環依存を作らない。共有したいものは共有パッケージへ寄せる

## 命名・用語

- コード・ドキュメントの語彙は `docs/CONCEPT.md` に合わせる:
  ワールドステート、ChainAdapter、チェーンプロファイル、ワークベンチ、
  ロギングプロキシ、A層〜D層
- 可視化階層の略号は **A層〜D層**。「L1」「L2」はブロックチェーン用語
  （基盤チェーン/ロールアップ）と衝突するため階層名には使わない
- UI 文言・用語集は日本語・英語の2言語対応（`{ja, en}` 形式）を前提に書く

## 開発ルール

- コミットメッセージは Conventional Commits 形式
  （`.githooks/commit-msg` で強制。`docs/CONTRIBUTING.md` 参照）
- ソースコード・依存関係を変更したら sync-docs スキルで
  `docs/` が実装を反映しているか確認する
- 各 Phase（`docs/CONCEPT.md` のロードマップ）が単体で「動くデモ」に
  なることを優先し、先の Phase のための先回り実装をしない
- GitHub（`morichikawa/chainviz`、private）で Issue 管理する。
  `docs/PLAN.md` の**チェックボックス1行 = Issue 1つ**の粒度で、着手する
  ステップに入るタイミングでそのステップの全チェックボックス分をまとめて
  Issue化する（ステップ単位でまとめて1 Issue にはしない）。ステップは
  GitHub の milestone に対応させる。担当パッケージに応じて
  `node-env` / `collector` / `frontend` / `shared` ラベルを付ける。
  チェックボックスにチェックを付けたら対応する Issue も閉じ、
  `docs/PLAN.md` にその Issue 番号へのリンクを併記する
- **Issue ごとにブランチを切る**。`main` 上で直接作業しない。ブランチ名は
  `issue-<番号>-<内容を表す短い英語スラッグ>`（例: `issue-1-genesis-pos-net`）
- **1つの変更内容 = 1コミット**に分ける。異なる関心事（設定変更とロジック
  変更、複数ファイルにまたがる別々の目的の変更など）を1つのコミットに
  まとめない
- 作業が終わったら PR を作成し、本文に `Closes #<番号>` を含めて Issue と
  連動させる。`chainviz-reviewer` と `chainviz-qa` のレビューを経てから
  `main` にマージする。マージは squash にせず、分けたコミットの履歴を
  保持する（通常のマージコミット、または rebase マージ）
- CI（`.github/workflows/ci.yml`）が push のたびに lint / build / test を
  実行する。ローカルで `pnpm lint && pnpm build && pnpm test` が通ることを
  確認してから push する
- タスク完了時は `docs/WORKLOG.md` に作業記録を追記する（何を・なぜ・どう
  実施したか、次の担当が知っておくべき注意点）。`docs/PLAN.md` のチェック
  ボックスは進捗管理用であり、経緯の記録は WORKLOG.md の役割
