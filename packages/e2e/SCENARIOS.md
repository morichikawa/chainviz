# E2E シナリオカタログ

E2E（結合）テストのシナリオを自然言語（箇条書き）で管理するカタログ。
このファイルが**シナリオの正**であり、テストコードはこれを実装する
（記法・実装規約は `docs/ARCHITECTURE.md` §8.4 を参照）。

- UI 層シナリオ（`UI-*`）: Playwright（chromium）で frontend を実際に
  操作して検証する。`src/ui/*.spec.ts` に実装し、`test()` のタイトルを
  「`<シナリオID>: <タイトル>`」、各箇条書きを `test.step()` にする
- プロトコル層シナリオ（`PROTO-*`）: WebSocket クライアント（`ws`）で
  collector を直接検証する。既存の `src/*.test.ts` が実装
- 新しい UI 機能を実装するときは、このカタログにシナリオを追記してから
  UI 層テストを実装する（`docs/ARCHITECTURE.md` §8.4 の運用ルール）

ステータス凡例: `済` = 実装済み / `予` = 未実装（実装 Issue 待ち） /
`保` = 前提機能の実装待ち。

## 2. UI 層シナリオ（Playwright）

前提（全シナリオ共通）: `profiles/ethereum` の Docker スタックが稼働し
チェーンが進行している。collector（UI 層専用ポート）と vite dev server が
起動済み（`playwright.config.ts` の globalSetup / webServer が行う）。

### 接続・初期表示（UI-CONN）

#### UI-CONN-01 collector に接続すると接続済みバッジが表示される

- 前提
  - collector が起動している
- 操作
  - ブラウザで frontend を開く
- 確認
  - 接続ステータスバッジが「接続済み」になる
  - 「モックデータ」の表記が**無い**（実 collector に接続している）

### A層: インフラ表示（UI-A）

#### UI-A-01 compose の全ノードとワークベンチがカード表示される

- 前提
  - compose 起動の 6 ノード（reth1/reth2/beacon1/beacon2/validator1/
    validator2）とワークベンチが稼働している
- 操作
  - frontend を開き、スナップショットの反映を待つ
- 確認
  - 7 枚のカード（`infra-card-<stableId>`）が表示される

  - ワークベンチのカードには操作ボタンがあり、ノードのカードには無い
  - ブートノードのカードにブートノードバッジが表示される
    （非ブートノードのカードには表示されない）

#### UI-B-05 チェーンリボンにブロックが連なって表示される（Issue #298）

- 前提
  - チェーンが進行し続けている（数秒ごとに新ブロック）
- 操作
  - frontend を開き、チェーンリボンカード（`chain-ribbon-card`）が
    表示されるまで待つ
- 確認
  - チェーンリボンカードにタイル（`chain-ribbon-tile-<hash>`）が
    1件以上表示される
  - しばらく待つと新しいタイルが右端に追加され、ヘッダの最新ブロック番号
    （`chain-ribbon-latest`）が増える
  - タイルにホバーするとポップオーバーが開き、ブロック番号・ハッシュ・
    親ブロック・時刻が表示される
  - マウスをタイルからポップオーバーの「親ブロック」行まで実際に移動
    させても、ポップオーバーは（遅延クローズの猶予を超えて静止しても）
    開いたままで、直前タイルの強調が持続する（Issue #351。以前は
    タイル→ポップオーバー間の隙間を経由するとポップオーバーごと閉じて
    しまい、強調をほぼ確認できなかった）
  - その場からマウスを離すと、強調・ポップオーバーの両方が消える

#### UI-B-06 チェーンリボンのタイルホバーでウォレット/コントラクトカードが連動して光る（Issue #298 第2段階）

- 前提

    ハイライトクラス（`infra-card--ribbon-highlight`）が付く
  - ホバーを外すとハイライトが消える

### 異常系（UI-ERR）

#### UI-ERR-01 collector が落ちると切断バッジになる `済`

- 前提
  - frontend が collector に接続済み（「接続済み」バッジ）
- 操作
  - （テストハーネスが）collector プロセスを停止する
- 確認
  - 接続ステータスバッジが「切断」に変わる
- 操作
  - collector を再起動し、ページをリロードする
- 確認
  - 「接続済み」に戻り、カード一式が再表示される
  - 備考: frontend の WebSocket クライアントは自動再接続を持たない
    （ARCHITECTURE.md「未確定のまま残す項目」）ため、復帰はリロードで
    確認する

#### UI-ERR-02 collector 停止中の追加操作はエラーが利用者に伝わる `済`

- 前提
  - collector が停止し「切断」バッジが出ている
- 操作
  - ツールバーの「ノード追加」ボタンを押す
- 確認
  - WebSocket が未接続でコマンドがそもそも送信できないため、ゴーストカードは
    1枚も作られず、即座にエラートーストで理由が利用者に伝わる
  - 備考: 当初の実装（`websocket/client.ts` の `sendCommand` が未接続でも
    `commandId` を発行してしまい、ゴーストカードだけが
    `GHOST_TIMEOUT_MS`（60秒）後に無言で消える）は
    [Issue #235](https://github.com/morichikawa/chainviz/issues/235)
    として起票し、`sendCommand` が未接続時に `undefined` を返すよう修正
    済み（`useCommands.ts` の `dispatch` がゴーストを作らず
    `describeCommandNotConnectedError` のトーストを即座に出す）。本項は
    その修正後の挙動を検証する

#### UI-ERR-03 定型操作フォームの不正入力は実行前に弾かれる `済`

- 操作
  - 送金フォームの金額に数値として解釈できない値（例: `abc`）を入力して
    実行を試みる
- 確認
  - バリデーションエラーが表示され、コマンドが送信されない
    （保留表示・トーストが出ない）
  - 備考: 原案では宛先に不正なアドレス（例: `0x123`）を入力する想定
    だったが、実機で確認したところ `TransferForm.tsx` の `canSubmit` は
    宛先が空でないことしか見ておらず、アドレス形式のバリデーションが
    無いためそのまま送信されてしまうことが判明した（Issue #209の
    DeployForm引数の同種の問題と同じ）。
    [Issue #236](https://github.com/morichikawa/chainviz/issues/236)
    として別途起票し、本シナリオの操作対象は「金額の不正な値」に
    差し替えた（金額側は `parseEtherToWei` によるバリデーションが
    実際に機能している）

#### UI-ERR-04 collector 側で失敗する操作はエラートーストで伝わる `済`

### 通信ログ（UI-LOG。Issue #317）

キャンバスに一瞬だけ現れる出来事（操作パルス・内部API活動パルス・ブロック
伝播・tx確定）を時系列ログとして遡れることを確認する（設計は
`docs/worklog/issue-317.md` 参照）。第1弾はフロントエンドのみで完結する
（既存 DiffEvent からの導出、collector・sharedの変更なし）。

#### UI-LOG-01 ツールバーのトグルボタン・Escでパネルが開閉する

- 操作
  - キャンバス左上ツールバーの「通信ログ」ボタンを押す
- 確認
  - サイドパネル（`side-panel`）が開き、`comms-log-view` が表示される
  - ボタンが押下状態（`aria-pressed="true"`）になる
- 操作
  - Esc キーを押す
- 確認
  - パネルが閉じる（`side-panel` が非表示になり、ボタンの `aria-pressed`
    も `false` に戻る）
- 操作
  - 再度ボタンを押して開き、もう一度同じボタンを押す
- 確認
  - トグル再押下でも閉じる

#### UI-LOG-02 送金操作の実行後に「操作（RPC）」エントリが記録される

- 前提
  - 通信ログパネルを開いている
- 操作
  - ワークベンチカードから送金操作を実行する（`support/operations.ts` の
    `submitTransfer`）
- 確認
  - しばらく待つと、操作（RPC）カテゴリのエントリ
    （`comms-log-entry[data-category="operation"]`）が新たに1件以上現れる
  - 内容にワークベンチ名 → 対象ノード名の主体表記とメソッド名
    （`eth_sendRawTransaction`）が含まれる
  - `eth_sendRawTransaction` を含むエントリに所要時間（ms）と成功表示
    （レスポンス観測。Issue #352）が含まれる。実環境の送金RPCは成功する
    前提で検証し、失敗ケースの決定的な検証はユニットテスト側で担保する
    （ここには含めない）

#### UI-LOG-03 ブロック進行で「ブロック受信」エントリが増える

- 前提
  - 通信ログパネルを開いている
- 操作
  - 次のブロックが生成されるまで待つ（チェーンは常時進行中）
- 確認
  - ブロックカテゴリのエントリ
    （`comms-log-entry[data-category="block"]`）の件数が増える
  - 新しいエントリが一覧の先頭（新しいものが上）に現れる

#### UI-LOG-04 カテゴリフィルタで該当カテゴリだけに絞られる

- 前提
  - 通信ログパネルを開いており、複数カテゴリのエントリが既に蓄積されている
    （ブロック進行・内部API観測はチェーン進行だけで決定的に蓄積される
    ため、このシナリオではワークベンチ操作を行わない。操作カテゴリの
    記録・検証は UI-LOG-02 が既にカバーしている）
- 操作
  - 「ブロック」以外の全カテゴリチップを押して off にする
- 確認
  - 表示されるエントリが全てブロックカテゴリになる
    （`comms-log-entry[data-category="block"]` 以外が0件）
- 操作
  - off にした「内部API」カテゴリチップを再度押して on に戻す
- 確認
  - 蓄積されていた内部APIカテゴリのエントリが再び表示される（フィルタは
    表示のみに作用し、蓄積自体には影響しないことの確認）

### サイドパネルの幅リサイズ（UI-PANEL。Issue #362）
