# Issue #216 作業記録

### 2026-07-10 Issue #216 beacon/rethのペア追加制約についての設計判断（調査・設計）

- 担当: designer
- ブランチ: issue-216-independent-el-cl-add
- 内容: 「addNode が reth（EL）+ beacon（CL）を必ずペアで追加するのは不自然
  では。独立して追加できる方が実態に合うのでは」というユーザーの疑問
  （Issue #216）について、実際の Ethereum ノード運用・現状設計の経緯・
  変更した場合の影響を調査し、対応方針を判断した。実装コードの変更はない。

#### 1. 実際の Ethereum ノード運用における EL/CL 構成の整理

- **The Merge（2022年）以降、フルノードは EL + CL の両方が必須**。EL
  （reth / geth 等）と CL（lighthouse / prysm 等）は Engine API
  （JWT 認証付きの内部 API）で接続され、CL が forkchoiceUpdated /
  newPayload で EL を駆動することでチェーンが進む。
- **EL:CL の接続は実質 1:1 が標準**。
  - CL 側: 主要 CL クライアントは接続先の execution endpoint を 1 つしか
    受け付けない（lighthouse は v3.0 で複数エンドポイント指定のサポートを
    意図的に削除し、単一エンドポイントのみとした）。
  - EL 側: 1 つの EL を複数の CL で共有する構成は各クライアントの
    ドキュメントで非推奨とされている。複数の CL がそれぞれ
    forkchoiceUpdated で「チェーンの先端はここ」と指示すると、EL への
    指示が競合しうるため。
- **「1:多・多:1」の柔軟な多重化が実在するのは VC（validator client）と
  BN（beacon node）の間の層**であり、EL:CL の層ではない。1 つの BN に
  多数のバリデーター鍵をぶら下げる（多:1）、冗長化のため 1 つの VC が
  複数の BN に接続する（1:多。lighthouse VC の `--beacon-nodes` 複数指定、
  Vouch 等）はいずれも VC:BN 層の話。chainviz の compose 構成でも
  validator1/validator2 は beacon1/beacon2 とは別サービスとして分かれており、
  この層の分離は既に表現されている。
- EL と CL を**独立して起動・停止できること自体は事実**（片方だけ再起動する
  運用は普通にある。chainviz でも Issue #43 で beacon 単独再起動を扱った）。
  ただしそれは一時的な状態であり、**定常状態として目指す形は常に
  「EL 1 : CL 1 のペア」**である。

結論として、「EL/CL を独立に追加でき、1:多・多:1 も組める」のが一般的、
という前提は EL:CL 層については当てはまらない。ユーザーの直観はおそらく
VC:BN 層の柔軟性（あるいは The Merge 以前の「EL 単体でフルノード」の
記憶）と混ざったものと考えられる。

#### 2. 現状のペア追加が意図的な設計であることの確認

「実装上の制約でたまたまそうなっている」のではなく、以下の複数箇所で
一貫して明文化された**意図的な設計判断**であることを確認した。

- `packages/collector/src/adapters/ethereum/node-lifecycle.ts` 冒頭コメント:
  「新規ノードは『バリデーターなしのフォロワー reth + beacon ペア』として
  追加する」「reth と beacon で同じ n を共有することで両者が同じ論理ノード
  として対応付く」（ステップ 5、Issue #34。`docs/worklog/issue-34-36.md`）。
- `docs/ARCHITECTURE.md` §7.6（Phase 5 / D層の UX 設計）: 内部リンク
  エッジ（beacon → reth）の常設本数を「reth+beacon のペア数」前提で設計し、
  「CL と EL が『1つの論理ノード』であることが見えない」ことを D層の
  設計動機の第一に挙げている。
- `glossary/ethereum/terms/d-internal.yaml` の `el-cl-separation`:
  「chainviz ではノードを追加すると必ず reth（EL）+ beacon（CL）の2枚の
  カードが対で現れ、内部リンクのエッジで結ばれる」と、**ペア追加の挙動
  そのものを EL/CL 分離の学習装置として用語定義に織り込んでいる**。
  この用語は内部リンクエッジのポップオーバー
  （`InternalLinkEdgePopover.tsx`）から参照でき、実装済み。

#### 3. 独立追加を可能にした場合に起きること（(b) 案の評価）

仮に EL 単独・CL 単独の追加を許すと、次の壊れ方をする。

- **EL（reth）単独**: The Merge 以降、新規ブロックの伝播は CL 側の
  ゴシップで行われ、EL はチェーン先端の更新指示（forkchoiceUpdated）を
  CL から受け取る。CL の無い reth は同期目標を持たず、キャンバス上では
  「同期中 / ブロック高 0」のまま動かないカードになる。これはステップ 5 の
  QA で実際に観測した「追従しない reth」（Issue #44 以前）と同じ見え方で、
  「各 Phase が単体で動くデモになる」原則（CLAUDE.md）に反する。
- **CL（beacon）単独**: lighthouse bn は The Merge 以降 execution endpoint の
  指定が必須で、EL なしでは正常起動できない。既存の reth（例: reth1）へ
  相乗りさせる「多:1」は、上記 1. のとおり実運用でも非推奨の構成であり、
  学習アプリがあえて見せる形として不適切。
- 実装面でも、`ManagedNode`（execution+consensus の組）・ノード番号 n の
  共有採番（172.28.1.n / 172.28.2.n）・removeNode のペア一括削除・
  frontend の内部リンクエッジ（drivesNodeId）と、collector / frontend /
  glossary の広範囲がペア前提で成立している。壊れた構成（機能しない
  カード）を作れるようにするために、この前提を崩す変更コストを払う
  学習上の価値は無い。

#### 4. 結論: (a) 現状維持

ペア追加は「学習・デモ用途のための単純化」ですらなく、**The Merge 以降の
実際の標準構成（EL 1 : CL 1 で 1 つの論理ノード）をそのまま反映した挙動**
である。独立追加を可能にすると、実運用でも組まない・機能しない構成を
作れるようになるだけで、学習支援という目的に対して逆効果。よって設計変更は
行わず、現状のペア追加を維持する。

- shared 型・collector・frontend への変更: なし。
- Issue #216 へは本メモの要約をコメントとして残し、ユーザーが後から
  経緯を読めるようにする（クローズ自体は統括の判断に委ねる）。

#### 決定事項・注意点

- **EL:CL = 1:1 のペア追加は今後も chainviz の前提としてよい**。
  ノード関連の機能（追加・削除・D層の内部リンク）はこの前提の上に
  設計・実装されており、変更しない。
- **VC:BN 層の柔軟性（1 VC : 複数 BN 等）は別の話**。将来もし
  「バリデーターの付け替え・冗長構成」を可視化したくなった場合は、
  EL:CL のペアを崩すのではなく VC を独立したエンティティとして扱う方向で
  設計する（現時点ではロードマップに無く、先回りしない）。
- **UX 上の残ギャップ**: 「なぜペアで追加されるのか」の説明は内部リンク
  エッジのポップオーバー（用語 `el-cl-separation`）に存在するが、追加
  操作を行うツールバー付近には無く、ユーザーが操作時点で疑問を持った
  こと自体がその証左。ノード追加ボタン付近に「reth + beacon のペアで
  追加されます」という一言を添える小さな改善は検討の余地があるため、
  別 Issue #251 として起票した（本 Issue の範囲では実装しない）。
