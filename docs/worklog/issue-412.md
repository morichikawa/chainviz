#### 1. 読んだドキュメントと分かったこと

- `docs/CONCEPT.md`「ロードマップ」: Phase 6（AA可視化）→ Phase 7以降
  （他チェーンプロファイル追加）が次に控えており、「攻撃手法の理解」は
  ロードマップのどこにも明示されていない。既存の学習用砂場（ハッシュ
  連結=Issue #401、署名検証=Issue #402。`docs/ARCHITECTURE.md` §15・§16）
  も同様にロードマップの外側の「発展」機能として扱われている。これに
  倣うのが自然という判断をIssue本文に記載した
- `docs/ARCHITECTURE.md` §9（フォーク判定・表現）・§10（チェーンリボン）:
  リオーグに関わる可視化は既に実装済み（Issue #296 / #298）。攻撃教育
  機能を作る場合、ゼロから作るのではなくこれらを流用できる可能性が高い
- `docs/ARCHITECTURE.md` §9.4・`docs/worklog/issue-296.md`（4節「フォーク
  誘発の試み」）: 設計時点で机上候補だった「ホスト側 iptables
  （DOCKER-USER チェーン）で beacon 間 P2P を DROP する」手順は、実際の
  QA検証で `sudo` がパスワードを要求し実行不能だったことが判明している。
  実チェーン上で意図的にネットワーク障害を注入する系統の案（後述の案2）
  は、この制約を踏まえた代替手段（コンテナへの `NET_ADMIN` capability
  付与など）の実現可否が実装着手前の検証事項になる、と明記した
- `docs/ARCHITECTURE.md` §15・§16（crypto-demo パターン）: サイドパネルの
  新規 kind・独立フォルダ・実チェーン非依存の疑似データ、という確立された
  パターンがある。攻撃手法のうち実チェーンでの再現が非現実的なもの
  （51%攻撃・ロングレンジ攻撃・eclipse攻撃）は、このパターンを踏襲する
  案が実装担当にとって最も迷いが少ないと判断した
- `docs/ARCHITECTURE.md` §4（`OperationEdge`・`runWorkbenchOperation`）:
  フロントランニング・メモリプールへの攻撃（issue本文にも言及あり）は、
  既存の複数ワークベンチからの競合tx送信・mempool可視化（Issue #84）の
  延長で表現できる見込みが高く、`packages/shared` の型変更なしで済む
  可能性がある

#### 2. 洗い出した3案（詳細はGitHub Issue #412本文を参照）

1. **案1: グロッサリー解説 + 既存可視化との紐付け**（教育コンテンツのみ、
   再現なし）。実装コストが最小・リスクがほぼ無いが、「見て触って理解する」
   というCONCEPT.mdの核心的な体験からは後退する
2. **案2: 実チェーン上での再現**（既存の観測・可視化機構を流用）。
   「本物のP2P・本物のfork choiceで学ぶ」という設計哲学に最も忠実だが、
   上記の sudo 制約のように環境依存のリスクが実測で既に一度顕在化している
3. **案3: フロント内で完結する独立シミュレーション砂場**（#401/#402と
   同じパターン）。実環境非依存でリスクが低く、51%攻撃・ロングレンジ攻撃・
   eclipse攻撃のような実チェーンで再現不能な概念もカバーできるが、
   トイモデルへの抽象化で「本物らしさ」が薄れるおそれがある

攻撃手法（51%攻撃・ダブルスペンド・リオーグ・フロントランニング・
ロングレンジ攻撃・eclipse攻撃・メモリプールへの攻撃全般）ごとに向き
不向きが大きく異なるため、「1案に決め打ち」よりも攻撃ごとの使い分けが
現実的という考察も添えた（比較表はIssue本文参照）。

#### 3. 意図的に決めなかったこと

- どの案（または組み合わせ）で進めるか
- ロードマップへの正式な組み込みか、#401/#402と同じ「発展」扱いか
- 案2を選ぶ場合の障害注入手段の実現可否（実測が必要）
- 対象とする攻撃手法の範囲（issue本文の「evm攻撃」「メモリプールに対する
  攻撃」を含め、どこまでを今回のスコープに含めるか）

いずれもユーザー・統括の判断が必要な分岐点のため、GitHub Issue本文に
「決まっていない点」として明記し、実装には進んでいない。

#### 4. 次の担当が着手する際の補足（型の見取り図・参考程度）

以下は実装するものではなく、あくまで各案を選んだ場合に検討が必要になる
点の見取り図（Issueオーナーが案を選んだ後、改めて `chainviz-designer` が
正式な型設計をする前提）:

- 案2（実チェーン再現）でシナリオ開始/終了をGUI操作にする場合、
  `Command`（`packages/shared/src/protocol/`）に新規 action
  （例: `{ action: "startAttackScenario"; scenario: string }` /
  `{ action: "stopAttackScenario" }`）を追加する形が既存の
  `addNode`/`removeNode` と対称的で自然。ただし本当に新規 action が
  必要か（既存の `runWorkbenchOperation` の組み合わせで足りるケースも
  ある。フロントランニングは恐らく後者）は攻撃ごとに見極めが要る
- 案3（独立シミュレーション砂場）は `packages/shared` の変更が原理的に
  不要（#401/#402と同型）。フロントの `crypto-demo/` と並ぶ新規フォルダ
  （例: `attack-demo/`）とサイドパネルの新規 kind（例:
  `"attackDemo"`）を足す形になる見込み
- 案1（グロッサリー）は新規 YAML ファイル1枚（例:
  `glossary/ethereum/terms/security.yaml`）とアンカー追加のみで、
  スキーマ変更は不要
